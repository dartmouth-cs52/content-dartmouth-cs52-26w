Front-end Testing

In a nutshell, front-end testing is all about ensuring everything works correctly on the client side of an application. This includes visible components, functionality, layout, or other elements of user experience like loading speed. As you can imagine, there‚Äôs almost no limit to the number of tests you can write and it‚Äôs not possible to test against every possible error in the real world. This means that you need to scale your tests to fit the project.

Why Test?

As mentioned above, front-end testing is useful for a variety of reasons; it ensures that your application performs as expected and catches changes early.

  • Responsiveness and cross-browser compatibility: Testing the application on a variety of screen sizes to make sure that the layout and functionality do not break. Testing that everything works across different browsers is also important but we will not worry about that here.
  • Spotting performance issues: This is to make sure that everything loads fast enough and that the user has a smooth experience overall. The performance of a website depends on a large number of systems making testing in the front-end the only way to know if there‚Äôs an issue.
  • Testing 3rd party resources: These resources (such as fonts, images, or other services) may be critical to your website but are also outside of your direct control. Having tests in place to identify issues is important.

What to Test?

Here‚Äôs an overview of some common ways to actually perform these kinds of tests.

  • Unit tests test the smallest components or ‚Äúunits‚Äù of your website independently of anything else on the page. Performing checks to make sure that buttons work, animations function, carousels carousel, you get the idea.
  • Visual tests are tests that rely on the visual presentation of your website as it is displayed in the viewport. Here, you‚Äôre ensuring that the website looks how it should rather than is coded correctly. This means‚Äîfor example‚Äîthat you measure the position of an element with JavaScript rather than look at what CSS values you gave it.
  • Cross-browser testing is how we make sure that the website performs correctly across all browsers (or at least the ones we care about, looking at you Internet Explorer).
  • Accessibility testing is important and not to be overlooked. These include tests of screen reader compatibility, colorblind-friendly interfaces, and navigability.

There are more kinds of tests beyond these to meet the needs of the goals of the website and the evolving needs of users. As you can see, there‚Äôs a lot to cover but don‚Äôt worry, we‚Äôre going to be starting simple in this short assignment.

The default test is not very interesting, we can use patterns like this to test various things about the page. In this case, that a particular element has a specific property.

Now to test the flexbox functionality of the Day Trips component.

JavaScript
1234567891011
it('should display "day trips" as a flex row on ipad-2', () => {
  cy.viewport(sizes[1]);
  cy.visit('index.html');
  cy.get('.day-trips__list').should('have.css', 'flex-direction', 'row');
});

it('should display "day trips" as a flex column on iphone-x', () => {
  cy.viewport(sizes[0]);
  cy.visit('index.html');
  cy.get('.day-trips__list').should('have.css', 'flex-direction', 'column');
});

Let‚Äôs break this down. We first want to see that the default flex-direction is set to row to display each element horizontally and take advantage of the wider screen. We first set the size to 'ipad-2' via the list of sizes we made above and then visit our homepage. We can grab the Day Trips element with its class and use the .should() method to specify a CSS property.

Now we can set the size to iPhone and test that same property which should have changed from 'row' to 'column'. Pretty easy right? You can test the grids of other groups of elements for extra credit.

To test that the hamburger menu opens on a mobile screen size we first set the size and navigate to the homepage like before. The button that‚Äôs used in this implementation is actually a checkbox. Using Cypress we can force the checkbox to be activated and see what happens. By inspecting the element of the page we see that the menu is simply translated above the viewport with a transformation, which is a CSS property we can check.

JavaScript
12345678
it('should display nav on hamburger click in mobile view', () => {
  cy.viewport(sizes[0]);
  cy.visit('index.html');
  cy.get('.main-nav__mobile-nav-trigger').check({ force: true });
  cy.get('.main-nav__nav-links')
    .should('have.css', 'opacity', '1')
    .should('have.css', 'transform', 'matrix(1, 0, 0, 1, 0, 0)');
