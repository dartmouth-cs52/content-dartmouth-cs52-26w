Web Architectures
 

 

 

 

 

 
 
intro to web architectures
 
* lots of different ways to structure data flow and processing when structuring a modern web app * let's talk about a few of them and we'll keep expanding our map of understanding as we go
 
web architectures: static sites 
 
* an overview of different web architectures * we'll go over these more in-depth also * static sites * just files, transmitted on the network * what we've been doing so far
 
web architectures: server side rendered (SSR)
 
* server side rendering * server is running code * server constructs custom response for every page * uses template and results from database
 
web architectures: SPA
 
* single page app * 1 index html file that loads javascript * javascript then handles all content and urls * loads all content asynchronously and inserts into page * DOM manipulation + frontend framework * largely the architecture we'll be using * is **more like an application** than a series of pages -> loads app into browser * from then on does not reload that page - app runs and gets any further data it needs. no contact.html, blogpost1.html etc
 
web architectures: isomorphic
* isomorphic app * does both server side rendering * AND javascript client side content fetching and display * this is useful in a few cases where say you want bots to read your page better or for SEO, so we'll learn about it but not have to inmplement it. * we'll mostly build single page apps in this class, but there are easy ways to enable this more isomorphic rendering as react can be rendered server side!

Event Driven & Asynchronous  Programming
 

(and why javascript)

 

 

 

 
  • in many languages such as java, python, ruby, perl, C
    • when you run input/output (I/O) functions
    • they block
    • the main program has to wait
    • things like waiting for a file
    • or waiting for a network request to finish loading
 
the problem of non-blocking io
 
* javascript in your browser is basically single-threaded - does one thing at a time * but how can that work when you are presenting a user interface where user inputs can come in at any time anywhere? * and how can you make that work fast? * non-blocking - majority of functions that take time are asynch * nice animations here: https://www.youtube.com/watch?v=wB9tIg209-8
  • In your browser:
    • webpages are constantly making web requests
    • if that blocked the page would freeze
    • why doesn't it?
  • javascript is non-blocking!
    • how it do?!
 
the problem of non-blocking io
 
 
how would blocking vs non-blocking look like?
 
* alert is an exception and is blocking - why might it be bad to litter your code with alerts? * we set this request to explicitly be synchronous - maybe isn't clear from this example but if it took a long time... the whole page would be frozen * can you imagine if other things were blocking - say an image loading - so you couldn't click anything till it finished loading * or the network requests that a page makes in the background? if the page froze every time it had to do anything like fetch more items in a list?
  • application flow control driven by events/changes in state
  • central mechanism that listens for events
  • calls callback function once event is detected
 
event driven programming
 
 
* event driven programming is used when you have inputs that can come at any time - interrupts if you will * you may have seen this dealt with as a game while loop - did that happen in 1/10? * events trigger changes in state * in js there is a built in event loop
  • some functions take time
    • making network requests (talking to a server)
    • any io really
    • or setInterval / setTimeout
 
asynchronous javascript
 
 
* so far all the functions we have passed in have pretty much run instantly * this is not always the case! * setInterval
 
single-threaded event loop
 
 
* you've used this by registering an event with setTimeout/interval (doesn't have to be user input) * we register an event listener * so that on a particular event a function gets executed * in a particular context (remember this) * how might setTimeout actually be implemented in this system? * while loop checking timestamp. * so setTimeout technically is MIN time, not guaranteed time
 
event loop with threading (server side node.js)
 
* a queue of tasks to complete/check in with * depending on javascript engine can either be threaded of single - node is multi, browser is single
